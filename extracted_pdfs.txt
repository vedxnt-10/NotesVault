1. Purpose & Background
The Artificial Intelligence and Data Science (AIDS) Department at BMS College of Engineering needs a
single, always-available web portal where students can read and download semester-wise study notes
and previous year question papers (PYQs). Today this material is scattered across WhatsApp groups,
Google Drive links, and photocopies. The portal centralizes this into one clean, fast, mobile-friendly
website with no login friction for students, while giving the department (a single faculty admin) full control to
add subjects and upload content from a hidden, password-protected panel.
2. Goals
●
Give students instant, login-free access to notes and PYQs, organised by semester → subject → unit.
●
Give the admin (faculty) a simple private panel to add subjects and upload/replace/delete PDFs without
touching code.
●
Keep the interface minimal, clean, and fast — white background with blue accents, readable on mobile
and desktop.
●
Ensure the site stays fast and error-free even when hundreds of students access it at the same time
(e.g. night before an exam).
●
Ship an MVP that is simple to build, cheap to host, and easy for one person (the admin) to maintain
long-term.
3. Non-Goals (Out of Scope for v1)
●
No student accounts, sign-up, sign-in, or personalization (bookmarks, history, profiles).
●
No commenting, ratings, discussion forums, or social features.
●
No in-browser PDF editing or annotation — only view and download.
●
No native mobile app — a responsive website only.
●
No multi-department / multi-college support in v1 (structure should allow it later, see TRD, but it is not
built now).
●
No payment, ads, or monetisation of any kind.

5. Functional Requirements
5.1 Student-facing site (public, no login)
1
Landing / Home: Student opens the webapp and immediately sees a semester selector — tiles/cards
for Semester 3 through Semester 8 (6 total). No login screen, no splash ad.
2
Semester Dashboard: After choosing a semester, the student sees a grid/list of subject cards for that
semester only (subjects are whatever the admin has added — count varies per semester).
3
Subject Page: Choosing a subject opens a page with two clearly separated sections: (a) Notes,
broken into Unit 1 through Unit 5 (fixed structure, one expandable block per unit, each holding one or
more PDFs), and (b) PYQs, a single flat list of previous year question paper PDFs for that subject,
sorted newest upload first.
4
View / Download: Each PDF row has two actions only — View (opens PDF in an in-browser viewer /
new tab) and Download (saves the file). No edit, no delete, visible to students.
5
Empty states: If a unit or the PYQ section has no PDFs yet, show a friendly 'Not uploaded yet, check
back soon' message instead of a blank/broken section.
6
Navigation: Persistent breadcrumb / back path: Home › Semester X › Subject Y, so a student can jump
back a level in one tap.
7
Responsiveness: Full functionality on mobile (majority of traffic expected), tablet, and desktop widths.
5.2 Admin panel (hidden, password-protected)
1
Access: The admin panel lives at a secret, unpublished URL path (e.g. /portal-manage, exact
value configurable) that is not linked from anywhere on the public site and is not discoverable via
navigation, sitemap, or search-engine indexing.
2
Authentication: A single password gate (no username, no visible 'login' branding on the public site).
Correct password issues a short-lived session; wrong password shows a generic error with rate-limiting
to slow brute-force attempts.
3
Add Subject: Admin can add a new subject to a chosen semester by entering a subject name/code.
The 5 unit slots and the PYQ slot are created automatically — admin does not manually create these.
4
Upload Notes: For any subject, admin can upload one or more PDF files into any of the 5 unit slots.
5
Upload PYQs: Admin can upload one or more PDF files into that subject's PYQ list.
6
Manage content: Admin can rename a subject, delete a subject (with confirmation), delete/replace an
individual PDF, and reorder is not required (newest-first is automatic).
7
Feedback: Every upload/delete action shows a clear success or error toast (e.g. file too large, wrong
file type).
5.3 File handling rules
●
Only .pdf files are accepted for upload; any other type is rejected client-side and server-side with a
clear message.
●
A sensible per-file size cap (recommended 25–50 MB) is enforced, with the exact limit surfaced in the
upload UI.

2. High-Level Architecture
The system is a static, CDN-hosted React SPA that talks directly to Supabase's auto-generated
REST/RPC API and Storage API from the browser. There is no custom backend server to run or scale —
Supabase (a managed Postgres + object storage + edge function platform) absorbs that responsibility,
which is what makes the “hundreds of concurrent students” requirement cheap to satisfy: the frontend is
served from a CDN edge (near-infinite horizontal scale) and the backend is a managed multi-tenant
Postgres cluster with connection pooling (PgBouncer) built in.

4. Admin Authentication Flow
1
Admin navigates to the secret path and enters the single shared password in a plain form (no
username field, no 'Forgot password' link that would hint at accounts).
2
Password is sent to a Supabase Edge Function (verify-admin) which compares it against a hashed
secret stored in Supabase project secrets (never in frontend code).
3
On success, the function returns a short-lived signed JWT (e.g. 12-hour expiry) that the frontend stores
in memory + httpOnly-style cookie via the function response.
4
All subsequent write calls (insert subject, upload file, delete file) attach this token; Postgres RLS
policies check the token's custom claim before allowing writes.
5
On failure, return a generic 'Incorrect password' message; the Edge Function applies basic rate-limiting
(e.g. exponential backoff per IP) to blunt brute-force attempts.

6. Security Requirements
●
Row Level Security enabled on every table; anon (public) role gets SELECT only.
●
INSERT/UPDATE/DELETE on subjects and documents restricted to requests carrying a valid
admin JWT claim, verified inside the RLS policy.
●
Storage bucket pdfs: public read policy for GET; write/delete policy requires the same admin claim.
●
Admin secret path + password never appear in client-visible source maps or public repo — stored as
environment variables / Supabase secrets.
●
HTTPS enforced everywhere (default on Vercel/Netlify + Supabase).
●
File-type validation both client-side (input accept='.pdf') and server-side (check MIME/signature before
accepting upload).

1. Entity Overview
Four tables cover the entire data model: departments (future-proofing, only one row used in v1), subjects
(scoped to a semester), documents (a note or a PYQ file, scoped to a subject), and admin_sessions
(optional — only needed if session state is tracked server-side rather than via stateless JWT). PDF
binaries live in Supabase Storage; the database only stores metadata + storage paths.

5. Storage Bucket: pdfs
●
One public-read Supabase Storage bucket named pdfs.
●
Recommended path convention: sem{semester}/{subject_code}/unit{n}/{filename}.pdf
for notes, and sem{semester}/{subject_code}/pyq/{filename}.pdf for PYQs — keeps files
browsable even directly in the Supabase dashboard.
●
Public read policy allows anyone to GET a file by path (needed since students never authenticate);
INSERT/UPDATE/DELETE restricted to the admin claim.
6. Row Level Security (RLS) Policies
-- subjects & documents: public read
CREATE POLICY "Public read subjects" ON subjects
  FOR SELECT USING (true);
CREATE POLICY "Public read documents" ON documents
  FOR SELECT USING (true);
-- writes require the custom admin claim set in the signed JWT
CREATE POLICY "Admin write subjects" ON subjects
  FOR ALL USING (auth.jwt() ->> 'role' = 'admin')
  WITH CHECK (auth.jwt() ->> 'role' = 'admin');

-- storage.objects policies for bucket 'pdfs'
CREATE POLICY "Public read pdfs" ON storage.objects
  FOR SELECT USING (bucket_id = 'pdfs');
CREATE POLICY "Admin write pdfs" ON storage.objects
  FOR INSERT WITH CHECK (bucket_id = 'pdfs' AND auth.jwt() ->> 'role' = 'admin');
CREATE POLICY "Admin delete pdfs" ON storage.objects
  FOR DELETE USING (bucket_id = 'pdfs' AND auth.jwt() ->> 'role' = 'admin');

8. Data Volume Expectations
6 semesters × ~6–8 subjects each × (5 units + 1 PYQ list, a handful of PDFs per slot) yields roughly a few
hundred rows in documents and a similar number of small-to-medium PDF objects in storage — trivially
small for Postgres and well within Supabase free/pro tier storage limits.
